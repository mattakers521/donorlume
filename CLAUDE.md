# DonorLume — Project Context

DonorLume is a multi-tenant SaaS donor intelligence platform for nonprofits and charities, built by **Vibrant Causes, LLC**. It helps fundraisers discover prospects from IRS 990 filings, score lapsed donors for reactivation, and generate AI-powered personalized outreach.

- **Domain:** donorluma.com
- **Tagline:** Stop reacting. Start strategizing.
- **Target user:** Development Directors at mid-size nonprofits exporting from CRMs like Bloomerang, Salesforce NPSP, Little Green Light, Kindful.

### Schema-change workflow

When `prisma/schema.prisma` changes:
1. `npx prisma migrate dev --name <name>` — applies the migration to Neon **and** rewrites `node_modules/.prisma/client/` with the new typed delegates.
2. **Restart `npm run dev`.** Next.js (Turbopack) doesn't invalidate `node_modules/*` imports on file change, so the running Node process keeps the pre-migration Prisma client cached. Symptom of skipping this: `Cannot read properties of undefined (reading 'findMany')` on a new model.
3. Use `TaskStop <id>` + a fresh background `npm run dev` from this harness — `prisma generate` alone won't fix it.

### Why Prisma 6 (not 7)

`@latest` initially pulled in Prisma 7.8, which broke the spec's setup in two unrelated ways: it disallows `url = env(...)` in the schema's datasource block, and (more critically) its new default `engineType = "client"` requires a driver adapter passed to `new PrismaClient(...)` — `next build` fails with `PrismaClientConstructorValidationError: Using engine type "client" requires either "adapter" or "accelerateUrl"`. Setting `engineType = "library"` did not bypass it.

Rather than introduce `@prisma/adapter-pg` + `pg` purely to satisfy a Prisma 7 quirk, the dependency was downgraded to Prisma 6.19. The schema now matches Spec §3 byte-for-byte (datasource has `url = env("DATABASE_URL")`), and the Prisma 7-specific `prisma.config.ts` was deleted. If a future Prisma 7+ upgrade happens, expect to:

1. Move the connection URL out of the schema (back to `prisma.config.ts` or pass via the adapter).
2. Install a driver adapter (`@prisma/adapter-pg` for direct Postgres, `@prisma/adapter-neon` for Neon serverless, etc.) and wire it through `lib/prisma.ts`.

### Spec deviation: per-campaign overrides live on the draft, not the campaign

Spec §3's `OutreachCampaign` model has no `metadata` column, but the setup form lets the user override **orgName**, **mission**, **senderName**, **senderTitle** per campaign. Two options were viable: add a `metadata Json?` column to `OutreachCampaign` (schema migration) or persist the overrides per-draft in the existing `OutreachDraft.metadata` (which is in the spec). Chose the latter — no migration, and each draft becomes self-contained for regeneration. The trade-off is duplication: each draft stores the same `promptConfig` snapshot. For ~20 drafts per campaign that's negligible. If this scales to thousands per campaign, revisit.

### Anthropic API costs

- ANTHROPIC_API_KEY required (https://console.anthropic.com/settings/keys). Without it, the drafts endpoint returns 500 "ANTHROPIC_API_KEY is not set."
- Sonnet 4.6 is **$3/M input, $15/M output**. A typical donor draft prompt is ~400 input tokens + ~300 output ≈ **$0.0057/donor** uncached, dropping to ~$0.0042/donor on cached calls. 100 donors ≈ $0.50.
- Cache hit rate: confirm with `usage.cache_read_input_tokens` in server logs once we surface usage. The system prompt may be under the 2K-token minimum — that's fine, just no caching benefit.

### Status-priority rule

Multiple writers can race to update the same draft (open pixel, click redirect, webhook). The status field uses an implicit priority: `BOUNCED > REPLIED > OPENED > SENT > APPROVED > DRAFT`. Every writer either (a) sets a terminal timestamp (`bouncedAt` / `repliedAt`) without touching `status` if that would downgrade, or (b) promotes `status` only along the allowed direction (currently only `SENT → OPENED`). Counter columns use `{ increment: 1 }`. First-occurrence timestamps use `field: existing ?? new Date()`.

### Known caveats / what's not in this slice

- **`donorlume.com` isn't verified with Resend yet.** `EMAIL_FROM` defaults to `"DonorLume <onboarding@resend.dev>"` — that test address only delivers to your own Resend-account email. Real customer sends need a verified domain (SPF/DKIM/DMARC).
- **Webhook signatures.** `/api/webhooks/resend` accepts events with no signature verification — it logs a warning when `RESEND_WEBHOOK_SECRET` is unset. Full svix-based verification is a follow-up.
- **Bot prefetch inflates opens.** Apple Mail Privacy Protection and Gmail image proxies prefetch tracking pixels, so `openCount` over-counts. Clicks are reliable.
- **Localhost tracking URLs.** `NEXTAUTH_URL=http://localhost:3000` means pixels resolve only on your machine. Send to your own inbox or expose via ngrok to validate the full loop.
- **Not built** (next slice): bulk send `POST /api/outreach/campaigns/[id]/send`, timeline chart (no per-event open timestamps stored — `openedAt` is first-open only), tier restrictions, unsubscribe flow.

### Two React-19 / Next-16 quirks worth remembering

1. **Date.now() in render is impure-lint-flagged.** Capture once with `useState(() => Date.now())` and thread the stable value down. (Hit this in `scored-view.tsx` for the months-ago column.)
2. **Date objects do serialize across the RSC→Client boundary** in Next.js 16, so passing a `Donor` row with `lastGiftDate: Date | null` directly to a client component works. Just remember to handle the `Date | string | null` shape in display formatters since the in-flight type after upload is `Date` but after a page hydrate it may already be a string from initial server payload.

| Spec route | Implementation |

The proxy routes don't currently require auth (they're read-only ProPublica data) — gate them behind `withOrg` if rate-limiting per session matters.

**Default ProspectType.** All ProPublica saves currently default to `FOUNDATION` since they're 501(c)(3) entities from 990 filings. Refine when we add type detection (e.g., look at NTEE prefix or sub-status).

### One pattern that bit me — passing icons across the server/client boundary

Lucide-react icons are React **components** (function references). React Server Components can't serialize functions across to Client Components. So this fails at runtime:

```tsx
// ❌ Server Component passing component ref to Client Component
<KpiCard Icon={Target} iconColor={C.amber} />
```

Fix: render the icon in the server component and pass the resulting **ReactElement** as a `ReactNode` prop. ReactElements are plain serializable objects.

```tsx
// ✅ Pass the rendered <Icon /> instead
<KpiCard icon={<Target size={22} color={C.amber} />} />
```

This applies to any Client Component receiving lucide icons from a server parent. Components that import lucide icons internally (e.g., `Sidebar` reads `NAV` from `lib/nav.ts`) are fine because the import happens client-side.

### Pricing + 14-day trial model (updated 2026-05)
- **Plans:** Starter $49/mo ($39 annual) · Growth $149/mo ($119 annual) · Scale $499/mo ($399 annual) · Managed custom. Annual prices are the *per-month equivalent when billed annually* (~20% savings vs monthly). No free tier — every org gets a 14-day trial instead.
- **Trial:** Stamped at org creation in `/api/auth/register` + `/api/auth/onboarding` as `trialEndsAt = createdAt + 14 days` and `stripeStatus = "trialing"`. **No Stripe interaction required to start a trial.**

## Key Architecture Rules

1. **Multi-tenant isolation:** every DB query MUST include `orgId` in WHERE. Spec Section 7 has the `withOrg` middleware pattern — use it on every API route.
2. **Org → User is many-to-many** via `OrgUser` join table with `Role` (OWNER/ADMIN/MEMBER/VIEWER).
3. **AI prompts are templated per-org**, not hard-coded — see Spec Section 5. Subject line is parsed from response with `^Subject:\s*(.+)` regex (matches prototype `donorluma-app.jsx:600`).
4. **RFM+ scoring** runs server-side at CSV upload. Algorithm is in Spec Section 6 and prototype `donorluma-app.jsx:506` (`scoreAll`). Scoring weights: Recency 30 / Frequency 25 / Monetary 25 / Tenure 20. Tiers: 80+ High, 55+ Medium, 30+ Low, else Cold.
5. **Lapsed threshold** is configurable per-org (default 12 months), stored in `OrgSettings.lapsedThresholdMonths`.

## Email-address convention

DonorLume uses two customer-facing addresses. Keep them strictly separated when writing copy or wiring `mailto:` links:

- **`support@donorlume.com`** — customer support: product help, account/auth issues, billing questions handled by humans, data exports/deletes, legal/privacy contact, in-product CRM feedback. Used in `/help`, the AI chat fallback ("if you don't know something, suggest support@"), `/settings/*` CRM-integration cards, the seat-limit login error, and the Terms / Privacy markdown.
- **`hello@donorlume.com`** — general inquiries: sales, demo bookings, partnership/press, Managed-plan Enterprise contact, landing-page contact form, footer "Connect" link.

The landing contact form (`/api/contact`) and the "Book a Demo" CTA both target `hello@`. If a new flow needs a contact email, choose by intent — anything a logged-in customer would expect a fast response on → `support@`; anything a stranger would send → `hello@`.
